# Inscription v0.19 tooling specification

Inscription v0.19 keeps the v0.18 source language unchanged and adds deterministic host-integration metadata artifacts to the CLI:

```text
inscription compile SOURCE --emit interface-json
inscription compile SOURCE --emit c-header
```

This is a tooling-only sprint. Source parsing, type checking, source MLIR generation, lowering, optimization presets, object emission, executable emission, and `run` remain compatible with v0.18.

## Interface JSON emission

`compile --emit interface-json` emits deterministic JSON describing the loaded Inscription interface. The artifact may be written to stdout or to `-o OUTPUT`.

The mode supports `--module-root DIR`, `--runtime-checks`, `--opt-level none|basic|aggressive`, `-O0`/`-O1`/`-O2`, `--save-temps DIR`, and `--verify`. Runtime checks and optimization levels do not affect the JSON output. `--verify` may run the existing MLIR verification pipeline, but the emitted artifact remains JSON.

Interface JSON does not require a root `main`, does not run the program, and does not require LLVM/MLIR tools unless verification is requested.

The top-level format is:

```json
{
  "format": "inscription-interface-v1",
  "source": "main.ins",
  "module_root": ".",
  "root_module": null,
  "modules": []
}
```

Paths are relative to the root source directory where possible. The emitter avoids timestamps, absolute paths, usernames, hostnames, git hashes, and other nondeterministic metadata.

## Interface JSON contents

Each module entry includes:

- `name`, or `null` for an unmoduled root file
- `path`
- `imports`
- `constants`
- `records`
- `layout_records`
- `exports`
- `externs`

Constants include name, scalar type, and evaluated JSON value. Boolean constants use JSON booleans; integer constants use JSON numbers.

Value records include declaration-order scalar fields. Layout records include kind, size, alignment, field offsets, field byte sizes, field alignments, and padding byte offsets. Natural and packed layout metadata follows the v0.8 layout algorithm.

Exported and extern phrases include phrase display text, kind (`gives` or `does`), external symbol, declaration-order parameters, and return type or `null` for `does` phrases. Interface JSON includes extern declarations because they are part of the compilation interface.

## C header emission

`compile --emit c-header` emits a deterministic C header for exported scalar phrases. The artifact may be written to stdout or to `-o OUTPUT`.

C header emission does not require a root `main`, does not run the program, and does not require LLVM/MLIR tools unless verification is requested. Runtime checks and optimization levels do not affect the header output.

The header starts with:

```c
/*
 * Generated by Inscription.
 * Interface format: inscription-c-header-v1.
 * This header includes exported scalar phrases only.
 */
#pragma once
#include <stdint.h>

#ifdef __cplusplus
extern "C" {
#endif
```

and ends with the matching C++ guard close.

C headers include exported phrases only. Extern phrase declarations are intentionally omitted because they are requirements of the Inscription compilation unit, not functions provided by it.

## C header type support

v0.19 C header generation supports this conservative exported ABI subset:

```text
i32 -> int32_t
u32 -> uint32_t
i64 -> int64_t
u64 -> uint64_t
```

`does` exports map to `void` return type. No-parameter exports use `(void)`. Parameter names are deterministic generated names: `arg0`, `arg1`, `arg2`, and so on.

If an exported phrase uses `i1`, `i8`, `u8`, `i16`, or `u16`, C header emission fails. The restriction applies only to `--emit c-header`; exported phrase lowering still supports every scalar type.

Exported symbols must be C identifiers matching `[A-Za-z_][A-Za-z0-9_]*`. Interface JSON still supports dotted or otherwise non-C exported symbols, but C header emission rejects them. v0.19 does not emit compiler-specific `asm("symbol")` annotations.

## Save temps

Interface JSON and C header modes do not create saved temps unless `--verify` requests the MLIR verification pipeline. When verification is requested with `--save-temps DIR`, the existing MLIR temp behavior is used for the verification stages.

## Non-goals

Inscription v0.19 does not add source syntax changes, C ABI attributes, C header support for `i1`/`i8`/`u8`/`i16`/`u16`, record C structs, layout-record C structs, buffer/view C ABI, pointer parameters, string literals, include guards derived from source paths, timestamps, build IDs, package manifests, header installation, C caller compilation, linker flags, library search paths, a standard library, macros, generics, or custom MLIR dialects.
